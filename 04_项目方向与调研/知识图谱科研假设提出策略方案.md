## 零、工作汇总

本阶段工作的核心目标是设计一套基于医学知识图谱的科研假设提出策略：

```text
已有医学知识图谱
  ↓
发现潜在弱关联 / 未知关联
  ↓
提取可解释多跳证据路径
  ↓
生成结构化科研假设
  ↓
从新颖性、机制合理性和可验证性角度排序
```

已完成的工作包括：

| 工作 | 文件 | 结论 |
|------|------|------|
| K-Paths 框架复现 | `K-Paths_复现_Worklog.md` | K-Paths 能增强 LLM 在药物重定位关系分类中的表现 |
| K-Paths API 适配 | `K-Paths/llm/llm_inference_api.py`、`K-Paths/MAC_API_RUN.md` | 已将本地 vLLM 推理改为 OpenAI 兼容 API，适配 macOS 运行 |
| K-Paths 数据理解 | `医学知识图谱转K-Paths数据.md` | 明确自有医学 KG 转为 K-Paths 输入所需的数据契约 |
| DREAMwalk 调研 | `DREAMwalk_调研笔记_候选关联生成与KPaths结合策略.md` | DREAMwalk 可补齐 K-Paths 前置的候选关联生成环节 |
| 实验结果核验 | `K-Paths/outputs/*/predictions.csv` | 已复核 Qwen2.5-14B/32B Base 与 K-Paths 指标 |

总体结论：

**K-Paths 适合作为科研假设提出系统中的“路径证据检索与解释模块”；候选关联生成器经过初步调研后，暂定复现 DREAMwalk 并观察其在医学知识图谱上的候选生成效果。若效果可用，再将 DREAMwalk 输出的候选实体对接入 K-Paths，形成“候选关联生成 → 路径证据检索 → 科研假设生成”的流程。**

## 一、总体目标

### 1.1 目标任务

本方案面向以下任务：

```text
基于已有医学知识图谱，探索弱关联或未知关联，并将这些关联转化为可解释、可验证的科研假设。
```

第一阶段建议聚焦药物重定位场景：

```text
Drug - Disease 潜在治疗关联发现
```

后续可扩展到：

```text
Gene - Disease
Pathway - Disease
Biomarker - Disease
Drug - Gene - Disease 机制假设
```

### 1.2 核心问题拆解

| 问题 | 对应模块 |
|------|----------|
| 哪些实体对值得研究？ | Candidate Relation Generator |
| 这些实体对为什么可能有关？ | Graph Evidence Retriever |
| 如何把路径证据变成科研问题？ | Graph-to-Hypothesis Generator |
| 哪些假设更值得优先验证？ | Hypothesis Evaluator / Ranker |

### 1.3 当前方法边界

K-Paths 不是完整的假设发现系统。它的输入是候选实体对，输出是实体对之间的多跳路径证据。

```text
K-Paths:
  candidate pair + KG
    → K 条多跳路径证据
```

因此完整系统必须补充前后两个环节：

```text
候选未知关联生成
  → K-Paths 路径证据检索
  → LLM 科研假设生成
  → SuperJudge / 专家评价排序
```

## 二、总体架构

### 2.1 系统流程

```text
医学知识图谱
  │
  ├─ Stage 1: Candidate Relation Generator
  │    └─ 初步复现 DREAMwalk，观察候选关联生成效果
  │
  ├─ Stage 2: Known Relation Filter
  │    └─ 过滤已知直接关系和答案泄漏边
  │
  ├─ Stage 3: Graph Evidence Retriever
  │    └─ K-Paths 抽取多跳机制路径
  │
  ├─ Stage 4: Graph-to-Hypothesis Generator
  │    └─ LLM 将路径转为研究问题、机制解释、验证方案
  │
  └─ Stage 5: Hypothesis Evaluator / Ranker
       ├─ 新颖性
       ├─ 证据一致性
       ├─ 机制合理性
       └─ 可验证性
```

### 2.2 模块定位

| 模块 | 推荐方法 | 输入 | 输出 |
|------|----------|------|------|
| Candidate Relation Generator | DREAMwalk 复现观察 | biomedical KG | Top-K candidate pairs |
| Known Relation Filter | 规则 + 文献/已知边过滤 | candidate pairs | unknown candidate pairs |
| Graph Evidence Retriever | K-Paths | candidate pair + KG | `all_paths` / `all_paths_str` / `path_str` |
| Graph-to-Hypothesis Generator | LLM / 模板 | candidate pair + evidence paths | 结构化科研假设 |
| Hypothesis Evaluator | SuperJudge / 专家 | hypothesis package | 排序和评价理由 |

### 2.3 最小可行 Demo

建议优先做 Alzheimer’s disease 药物重定位 demo：

```text
输入目标疾病:
  Alzheimer's disease

DREAMwalk:
  输出 Top-K candidate drugs

Known Relation Filter:
  去掉已知 treats / palliates 关系

K-Paths:
  为 Drug X - Alzheimer's disease 检索机制路径

LLM:
  生成假设:
    Drug X 是否可能通过 Gene A / Pathway B 影响 Alzheimer's disease？

Evaluator:
  从新颖性、机制合理性、可验证性排序
```

## 三、核心模块设计

### 3.1 Candidate Relation Generator

#### 目标

从已有医学知识图谱中自动发现可能存在但尚未确认的候选实体对。

第一阶段聚焦：

```text
Drug - Disease candidate association
```

#### 初步方案：复现 DREAMwalk

DREAMwalk 来自 Nature Communications 2023，面向 biomedical KG 上的 drug-disease association prediction。当前对候选关联生成器的调研先不展开多模型对比，初步定为复现 DREAMwalk，并观察它能否从医学 KG 中输出有价值的 drug-disease candidate pairs。

核心思想：

```text
drug-gene-disease 多层图谱
  → semantic information-guided random walk
  → 学习 drug / disease embedding
  → 预测潜在 drug-disease association
```

输出形式：

```json
{
  "target_disease": "Alzheimer's disease",
  "candidate_drugs": [
    {
      "drug": "Drug A",
      "score": 0.92
    },
    {
      "drug": "Drug B",
      "score": 0.88
    }
  ]
}
```

DREAMwalk 的价值是补齐 K-Paths 前置环节：

```text
DREAMwalk 负责“找可能有关系的实体对”
K-Paths 负责“解释它们为什么可能有关系”
```

### 3.2 Known Relation Filter

候选关联生成后，需要过滤已知强关系，避免把已有知识包装成“新假设”。

过滤对象包括：

- KG 中已有直接 `treats` / `indication` / `contraindication` 边；
- 训练集中已出现的阳性关系；
- 文献或数据库中已明确确认的 drug-disease 关系；
- K-Paths 抽取时会造成答案泄漏的目标边。

示例：

```text
如果 Donepezil treats Alzheimer's disease 已知，
则 Donepezil - Alzheimer's disease 不作为未知假设输出。
```

保留对象：

```text
无直接强关系，
但存在 2 至 3 跳机制路径，
且候选生成模型给出较高关联分数的实体对。
```

### 3.3 Graph Evidence Retriever

该模块由 K-Paths 承担。

输入：

```text
candidate pair + augmented biomedical KG
```

输出：

| 字段 | 含义 |
|------|------|
| `all_paths` | 节点 ID 和关系 ID 构成的结构化路径 |
| `all_paths_str` | 带实体名称、类型和关系模板的路径 |
| `path_str` | 可直接输入 LLM 的自然语言证据 |

默认路径参数：

```text
max_length = 3
max_paths = 10
```

路径示例：

```text
Donepezil (Compound) binds CYP2D6 (Gene)
and CYP2D6 (Gene) is associated with Alzheimer's disease (Disease)
```

注意：K-Paths 的路径证据不等同于因果证据。尤其是 gene-disease association、anatomy localization、compound resemblance 等关系，可能只是相关性或共现证据，需要在假设评价阶段单独判断证据强度。

### 3.4 Graph-to-Hypothesis Generator

LLM 输入：

```text
candidate pair
DREAMwalk association score
K-Paths evidence paths
known relation filtering result
```

LLM 输出建议采用结构化格式：

| 字段 | 内容 |
|------|------|
| `research_question` | 研究问题 |
| `hypothesis` | 科研假设 |
| `candidate_relation` | 推测的新关系 |
| `evidence_paths` | 支持路径 |
| `mechanism` | 可能机制 |
| `novelty_rationale` | 新颖性说明 |
| `counter_evidence` | 可能反证或不确定性 |
| `validation_plan` | 实验或数据分析验证方案 |

示例：

```text
研究问题：
Drug X 是否可能通过 Gene A / Pathway B 影响 Alzheimer's disease？

科研假设：
Drug X 可能通过调控 Gene A 参与的 Pathway B，影响阿尔茨海默病的发生或进展。

验证方案：
在 AD 细胞或动物模型中检测 Drug X 对 Gene A 表达、Pathway B 活性和 AD 相关表型的影响。
```

### 3.5 Hypothesis Evaluator / Ranker

第一版可使用规则排序，后续接入 SuperJudge 或专家打分。

候选排序指标：

```text
Final Score =
  α * association_score
+ β * path_evidence_score
+ γ * novelty_score
+ δ * verifiability_score
```

第一版规则：

1. DREAMwalk 关联分数高；
2. 没有已知直接治疗关系；
3. K-Paths 找到多条独立机制路径；
4. 路径经过 Gene / Protein / Pathway 等机制节点；
5. 关系方向与机制解释一致；
6. 文献共同出现较少但路径证据充分；
7. 能设计明确实验或数据分析验证。

## 四、数据契约

### 4.1 医学 KG 输入

自有医学知识图谱需要提供三类基础信息：

```text
实体
关系
三元组
```

推荐文件：

| 文件 | 内容 |
|------|------|
| `node2id.json` | 业务实体 ID 到整数节点 ID 的映射 |
| `id_to_name_mapping.json` | 节点 ID 对应实体名称和类型 |
| `relations.json` | 关系 ID、方向和自然语言模板 |
| `BKG_file.txt` | 整数三元组 |
| `train/test.json` | 待预测实体对及评估标签 |
| `known_pairs.csv` | 已知强关系，用于过滤已有结论 |

### 4.2 实体映射

```json
{
  "DB00001": 101,
  "HGNC:11998": 5001,
  "DOID:10652": 9001
}
```

要求：

- 每个药物、疾病、基因等实体必须有唯一整数 ID；
- 业务实体必须能映射到图节点；
- 同义词、别名、外部数据库 ID 需要提前完成归一化。

### 4.3 关系映射

```json
{
  "6": "{u} (Drug) binds {v} (Gene)",
  "44": "{u} (Gene) is associated with {v} (Disease)"
}
```

要求：

- 每种关系必须有整数 ID；
- 每种关系必须有明确方向；
- 每种关系必须能转换成自然语言模板；
- 如果使用反向边，需要准备反向关系语义。

### 4.4 图三元组

K-Paths 原代码使用的物理列顺序是：

```text
source_node_id target_node_id relation_id
```

示例：

```text
101 5001 6
5001 9001 44
```

注意：这里不是 `source relation target`。

### 4.5 K-Paths 路径数据格式

当前复现使用的数据文件：

```text
K-Paths/data/paths/pharmaDB_test_add_reverse.json
```

该文件实际是 JSON Lines：

```text
每行一个 JSON object
```

样本数量：

```text
252
```

标签分布：

| 原始标签 | 归一化展示 | 数量 |
|----------|------------|-----:|
| `Disease-modifying` | Disease-modifying | 109 |
| `Palliates` | Palliates | 77 |
| `Non indications` | Non-indication | 66 |

关键字段：

| 字段 | 含义 |
|------|------|
| `drug1_db` | 药物外部 ID |
| `drug2_db` | 疾病外部 ID |
| `drug1_id` | 药物整数节点 ID |
| `drug2_id` | 疾病整数节点 ID |
| `drug_name` | 药物名称 |
| `disease_name` | 疾病名称 |
| `label` | 真实标签 |
| `label_idx` | 标签 ID |
| `all_paths` | ID 路径 |
| `all_paths_str` | 名称和关系模板路径 |
| `path_str` | LLM 输入路径证据 |

### 4.6 防止答案泄漏

抽取路径前必须删除待预测实体之间的目标答案边。

例如要预测：

```text
Drug A treats Disease C
```

图中不能保留该直接边，但可以保留间接证据：

```text
Drug A binds Gene B
Gene B is associated with Disease C
```

## 五、K-Paths 复现结果

### 5.1 实验设置

| 项 | 设置 |
|----|------|
| 任务 | 药物-疾病治疗关系分类 |
| 数据集 | PharmacotherapyDB |
| 样本数 | 252 |
| Base | 仅输入问题、候选标签和模型自身知识 |
| K-Paths | 在 Base 基础上加入最多 10 条、最长 3 跳的 KG 路径 |
| 模型 | Qwen2.5-14B-Instruct / Qwen2.5-32B-Instruct |
| 推理方式 | 硅基流动 OpenAI 兼容 API |

### 5.2 指标结果

已重新运行 `K-Paths/llm/evaluate_llm_regex.py` 核对指标：

| 模型 | 设置 | Accuracy | Macro-F1 | Kappa |
|------|------|---------:|---------:|------:|
| Qwen2.5-14B | Base | 60.71 | 60.98 | 42.23 |
| Qwen2.5-14B | K-Paths | **65.48** | **63.02** | **47.56** |
| 14B 增益 | | **+4.76** | **+2.04** | **+5.33** |
| Qwen2.5-32B | Base | 65.48 | 65.08 | 47.78 |
| Qwen2.5-32B | K-Paths | **70.63** | **65.70** | **52.83** |
| 32B 增益 | | **+5.16** | **+0.62** | **+5.05** |

与论文结果对照：

| 来源 | Accuracy | Macro-F1 | Kappa |
|------|---------:|---------:|------:|
| K-Paths 论文 | 71.83 | 67.83 | 55.10 |
| 本次 Qwen2.5-32B + K-Paths 复现 | 70.63 | 65.70 | 52.83 |
| 差距 | -1.20 | -2.13 | -2.27 |

### 5.3 结果解释

主要观察：

- 加入 K-Paths 后，14B 和 32B 的 Accuracy、Macro-F1、Kappa 均提升；
- Accuracy 和 Kappa 提升约 5 个百分点；
- 32B + K-Paths 结果接近论文报告值；
- K-Paths 对 `Disease-modifying` 和 `Palliates` 更有帮助；
- K-Paths 可能降低 `Non-indication` 类别表现，因为相关路径容易被 LLM 解释为治疗证据。

对科研假设发现的启示：

```text
KG 路径能增强模型判断，
但“存在生物学关联路径”不能自动等同于“存在治疗或因果关系”。
```

因此，在假设生成阶段必须加入：

- 路径关系类型判断；
- 关系方向判断；
- 证据强度评分；
- 反向证据检查；
- 新颖性和可验证性评价。

## 七、接入自有医学 KG 的实施路径

### Stage 0: 数据准备

输入：

```text
自有医学 KG 三元组
实体标准化表
关系标准化表
已知强关系表
```

输出：

```text
node2id.json
id_to_name_mapping.json
relations.json
BKG_file.txt
known_pairs.csv
```

### Stage 1: 候选关联生成

初步方案：

```text
复现 DREAMwalk，观察其候选 drug-disease 关联生成效果
```

输出：

```text
candidate_pair, association_score
```

### Stage 2: 已知关系过滤

过滤：

- KG 已有直接边；
- 训练集阳性边；
- 外部数据库已知关系；
- 文献中已明确确认的关系。

输出：

```text
unknown_candidate_pair, association_score
```

### Stage 3: K-Paths 路径抽取

将候选实体对转换为 K-Paths 输入格式：

```json
{
  "entity1_id": 101,
  "entity2_id": 9001,
  "entity1_name": "Drug A",
  "entity2_name": "Disease C"
}
```

输出：

```text
candidate_pair
all_paths
all_paths_str
path_str
```

### Stage 4: 假设生成

LLM 根据路径证据生成：

```text
研究问题
科研假设
机制解释
支持路径
不确定性
验证方案
```

### Stage 5: 假设评价

评价维度：

| 维度 | 说明 |
|------|------|
| 新颖性 | 是否缺少已知直接证据，是否不是旧结论复述 |
| 证据一致性 | 多条路径是否支持同一机制方向 |
| 机制合理性 | 路径关系是否能构成合理生物机制 |
| 可验证性 | 是否能设计实验或数据分析验证 |
| 风险 | 是否存在明显反证、毒性、适应症冲突 |

输出：

```text
ranked_hypotheses.json
```

---

## 八、评价方案

### 8.1 候选生成评价

可使用：

- 已知关系做 sanity check；
- temporal holdout 检查模型是否能预测后续出现的新关系；
- 人工抽查 Top-K candidate pairs 是否具有医学合理性和后续 K-Paths 检索价值。

### 8.2 路径证据评价

评价：

- 无路径样本比例；
- 每个候选平均路径数；
- 路径长度分布；
- 关系类型分布；
- 路径是否经过有机制意义的 Gene / Protein / Pathway；
- 是否存在高连接节点导致的泛化路径。

### 8.3 假设质量评价

建议使用四类评分：

| 评分 | 含义 |
|------|------|
| Novelty | 假设是否新颖 |
| Plausibility | 生物机制是否合理 |
| Evidence | KG 路径证据是否充分 |
| Testability | 是否可实验或数据验证 |

### 8.4 人工 / 专家复核

Top-K 假设需人工或专家复核：

- 路径是否医学上合理；
- 是否把相关性误读为因果；
- 是否忽略已知反证；
- 验证方案是否可实施。

## 九、风险与限制

### 9.1 K-Paths 限制

- K-Paths 不自动生成候选实体对；
- 默认偏向短路径和高连接节点；
- 多跳相关路径不等同于因果机制；
- 对 `Non-indication` 类别可能造成误判；
- 路径质量高度依赖 KG 覆盖、关系定义和实体标准化。

### 9.2 DREAMwalk 限制

- 主要面向 drug-disease association prediction；
- 输出 embedding 分数，机制解释较弱；
- 对 KG 结构和数据格式依赖较强；
- 泛化到 gene-disease、pathway-disease 等任务需要改造。

### 9.3 LLM 限制

- 可能过度解释弱路径；
- 可能把 KG 相关关系扩展为未经验证的因果关系；
- 需要强制输出反证和不确定性；
- 需要专家或规则系统进行二次评价。
